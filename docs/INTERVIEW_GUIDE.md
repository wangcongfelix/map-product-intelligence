# Interview Guide

## 30 秒介绍

这是一个面向地图与户外产品经理的 AI 用户声音与竞品信号分析工作台。它可以把应用评论、版本日志、社区反馈和竞品体验记录导入系统，通过 DeepSeek 或 Mock AI 做结构化预分析，再由产品经理人工审核，最后沉淀到信号总表，并生成图表和周报/月报风格的 Markdown 报告。

## 3 分钟项目介绍

### 背景

地图产品经理需要长期关注用户评论、竞品更新和社区反馈，但这些信息来源分散、格式混乱，很难直接形成产品判断。

### 问题

如果只靠人工整理，效率低且难以沉淀；如果直接让大模型总结，又容易出现结论不可追踪、分类口径不稳定和证据缺失的问题。

### 解决方案

我做了一个围绕产品经理工作流设计的 AI 工作台：资料导入后自动拆分文本，AI 先做模块、信号类型、影响程度和建议动作判断，产品经理再审核、修改或忽略，最终只把人工确认后的信号用于总表和报告。

### 核心流程

资料导入 → 文本拆分 → DeepSeek / Mock AI 预分析 → 人工审核 → 信号总表 → 图表分析 → Markdown 报告和 PNG 图表导出。

### AI 怎么用

AI 负责把非结构化文本变成结构化候选信号。Prompt 和 Schema 约束输出字段，Mock AI 负责无 Key 演示和回归测试，DeepSeek 负责正式预分析。

### 最终产出

产出不是聊天记录，而是可信的产品信号库、图表分析和公司周报/月报风格报告。

### 我的贡献

我完成了场景定义、核心工作流、AI 分析结构、DeepSeek 接入、Mock 兜底、人工审核、信号总表、报告中心和展示包装。

## 10 分钟深讲

### 产品目标

把地图产品经理日常面对的分散文本资料，转化成可审核、可沉淀、可汇报的产品信号。

### 用户角色

核心用户是地图、出行、户外产品经理，也包括需要做竞品分析和用户声音归纳的业务同学。

### 功能架构

前台分为 `/import`、`/review`、`/signals`、`/reports` 四个核心页面。服务层分为分析服务和报告服务，状态层使用轻量客户端 Store 和 LocalStorage 持久化。

### AI 结构化分析

AI 输出固定 Schema，包括模块、信号类型、影响程度、核心结论、原文证据、产品启示、建议动作和置信度。这样后续审核、统计和报告都可以依赖结构化数据。

### 人工审核

人工审核是项目的可信度核心。AI 的判断必须经过产品经理确认后，才能进入信号总表和报告中心。

### 报告中心

报告中心只读取正式信号，生成模块分布、来源分布、类型分布、影响程度分布，并输出适合复制到周报、月报或竞品分析文档的 Markdown 报告。

### 技术实现

项目使用 Next.js、React、TypeScript、Tailwind CSS、shadcn/ui、TanStack Query、DeepSeek OpenAI-compatible API、LocalStorage 持久化、Markdown 生成和 Canvas PNG 导出。

### 迭代和权衡

前期先用 Mock AI 建立流程，再接入 DeepSeek；先用 LocalStorage 降低复杂度，再保留未来接数据库和多用户的空间；报告先用确定性规则生成，避免 AI 直接编造正式报告。

### 项目价值

这个项目展示了 AI 产品不只是接一个聊天框，而是要嵌入真实业务流程，建立结构化输入、人工确认和可信输出。

## 面试官追问 Q&A

### 1. 为什么不直接让大模型生成报告？

因为报告是正式输出，直接生成容易出现幻觉和不可追溯结论。本项目先让 AI 生成候选信号，再由人确认，报告只基于确认后的数据生成。

### 2. 为什么需要人工审核？

产品判断涉及优先级、业务背景和竞品理解，AI 只能辅助初筛。人工审核能保证进入总表和报告的数据可信。

### 3. Mock AI 有什么价值？

Mock AI 让系统在没有 API Key 的情况下也能完整演示，并且能稳定做回归测试，降低开发成本。

### 4. 怎么保证 AI 输出结构稳定？

通过 Prompt 明确角色、字段和枚举范围，再用 Schema 校验输出。无效输出不能直接进入审核流。

### 5. DeepSeek Key 怎么保护？

Key 只放在本地 `.env.local`，由服务端 API 路由读取，客户端不暴露真实 Key，仓库只提交 `.env.example`。

### 6. 为什么用 LocalStorage，而不是数据库？

当前阶段目标是验证端到端产品闭环，LocalStorage 足够支撑单机演示和快速迭代。团队化使用时可以升级到数据库和账号体系。

### 7. 这个项目和普通评论分类有什么区别？

它不是单点分类工具，而是从资料导入、AI 预分析、人工审核、信号沉淀到报告输出的完整产品工作流。

### 8. 如果上线给团队用，下一步怎么扩展？

接入数据库、用户权限、团队协作、批量文件导入、报告模板、PDF/Word 导出、评估体系和线上部署。

### 9. 你个人贡献是什么，开源模板贡献是什么？

开源模板提供 Next.js + shadcn 的工程外壳和 UI 基础。我个人完成地图产品情报场景、业务流程、AI 分析结构、DeepSeek 接入、人工审核、信号总表、报告中心和项目包装。

### 10. 你如何评价这个项目的不足？

目前数据仍是本地存储，报告生成是确定性规则，缺少数据库、多用户、权限和线上部署。它更像一个完整 MVP，下一步要做团队化和生产化。
