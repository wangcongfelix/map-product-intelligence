# Project Review

## 项目背景

地图、出行和户外产品的用户声音来源非常分散，包括应用商城评论、版本更新日志、社区反馈、路线评论和竞品体验记录。产品经理需要从这些资料中持续识别问题、机会和竞品变化，但传统整理方式依赖人工阅读和手动归纳，效率低且难以复用。

Project002 的目标是把这些分散资料转化为可审核、可沉淀、可汇报的产品信号。

## 用户痛点

- 原始资料多、散、重复，难以持续跟踪。
- 用户评论和竞品动态通常没有统一结构。
- 直接让 AI 总结容易缺少证据和可复核链路。
- 产品经理需要的是能进入周报、月报和需求讨论的可信结论。

## 核心产品闭环

资料导入 → 文本拆分 → AI 预分析 → 人工审核 → 信号总表 → 图表统计 → 报告输出。

这个闭环把一次性的文本处理变成持续积累的产品信号库。

## 为什么设计 Human-in-the-loop

AI 适合做初步结构化和归因，但不适合直接成为最终判断。产品经理需要对模块、信号类型、影响程度和建议动作进行确认。

系统约束是：报告只基于人工确认后的 `reviewedAnalysis` 生成。这样可以降低误判和幻觉风险，也让每条结论都能回到原文证据。

## AI Prompt / Schema 的作用

Prompt 负责约束 AI 的分析角色、分类口径和输出要求。Schema 负责校验字段完整性和枚举合法性，保证 DeepSeek 或 Mock AI 的结果能进入稳定的审核流程。

结构化字段包括模块、信号类型、影响程度、核心结论、用户场景、原文证据、产品启示、建议动作和置信度。

## Mock AI 的价值

Mock AI 不是临时玩具，而是产品验证和工程回归的重要兜底：

- 无 API Key 时可完整演示流程。
- 降低开发和测试成本。
- 保证文本拆分、审核、信号总表和报告中心可以独立验证。
- 为 DeepSeek 接入前后提供稳定对照。

## 报告中心为什么只基于人工确认信号

报告是面向 leader、mentor 或业务组的正式输出，不应直接使用未经确认的 AI 判断。只读取人工确认后的信号，可以确保报告中的图表、结论和建议动作有明确来源。

## 关键迭代过程

- V1-1：完成资料导入、批量文本拆分、预览编辑和 LocalStorage 保存。
- V1-2：接入 Mock AI，建立待确认卡片、审核流程和信号总表。
- V1-3：接入 DeepSeek 真 AI，保留 Mock 兜底。
- V1-4：建设报告中心，加入图表、信号筛选和 Markdown 报告。
- V1-4.2：重构工作台 UI，优化导入、审核、信号和报告体验，并加入 PNG 图表导出。
- V1-4.3：完成默认主题、项目 Branding、默认 DeepSeek、报告格式和 Devtools 清理。

## 踩坑

- `.env.local` 不能提交，真实 API Key 只保留在本地。
- React Query Devtools 演示前必须移除，否则会暴露开发调试入口。
- 原模板仓库 remote 不能误 push，需要替换为项目仓库。
- 原开发仓库 push 出现 pack 问题，最终通过干净目录重新初始化发布解决，同时保留原目录完整历史。
